# 架构设计

## 当前分层

```text
App/UI
  负责脚本选择、状态展示、设置、悬浮控制、音量键控制、脚本 UI 绘制和 RootDaemon 生命周期

RootDaemon（uid=0）
  负责 Root 截图、输入注入、输入法系统切换和物理音量键监听，只接受本应用认证的回环 socket 请求

EngineService (:engine)
  负责脚本运行、停止、状态广播、HTTP 控制入口和到 RootDaemon 的客户端会话

libengine.so
  负责 Lua Runtime、Lua native 多线程、core/api 统一核心实现、C ABI 门面和当前截图核心

PC 工具 / VSCode
  平级直连 EngineHttpServer；VSCode 负责脚本开发，Qt 工具负责抓图、取色、图像分析和投影
```

## PC 工具边界

```text
ide/vscode-extension
  脚本运行、暂停、继续、停止、日志、打包，以及启动独立抓图工具

ide/xiaoyv-tools
  Qt 多标签图片工作区、取色、二值化、点阵字库、设备截图和投影

shared/native/image_core
  不依赖 Qt/Android 的颜色规则、二值化、点阵生成和完整点阵匹配
```

VSCode 与 `xiaoyv_tools.exe` 不互相中转。ADB 模式下二者分别管理自己的 forward，Qt 工具
使用动态本机端口；LAN 模式下分别连接设备 IP。Android `EngineHttpServer` 使用并发客户端
处理器，允许二者同时工作。

工具截图直接读取 `screen_api` 当前 RGBA8888 帧。投影只通过 `PUT /tool/image` 上传图片，
再调用 `viewer.openImage(fileId)` 启动主进程全屏 Activity。投影不进入脚本 API，也不在
`libengine.so` 建立额外显示状态。详细设计见 [PC 抓图取色工具](PC_抓图取色工具.md)。

固定设备能力统一在 `core/api`，Lua HostApi 通过 `system_c_api` 的 C ABI 调用它们。

各自的并发模型，不复用 Lua 线程对象。

## 当前边界规则

- App/UI 只负责界面、脚本文件选择、悬浮窗和状态展示。
- 音量键是 App 控制入口：Root 模式由 RootDaemon 直接读取 evdev，非 Root 模式由无障碍服务
  接收；两条路线最终都只向 `EngineService` 发送运行或停止 Intent。
- EngineService 只负责引擎进程、HTTP 服务、脚本控制和状态广播。
- 所有语言无关设备 API 的真实逻辑先进入 `libengine.so/core/api`，语言绑定层只做转换。
- Lua 多线程和动态 Java 对象属于语言运行时能力，实现在 `libengine.so/runtime/lua`。
- `system_c_api` 是 C ABI 门面，不承载具体业务逻辑。
- Java 桥只处理 Android 必须由 Java 完成的能力，例如 RootDaemon 生命周期、认证 socket 和平台桥接。
- `:engine` 不持有 `su` 会话；强停它不会关闭 RootDaemon，也不会在下一次脚本运行时重新请求 Root 授权。

## 点阵字库路径

```text
Lua / JS / Go / 插件
  -> engine_font* C ABI
  -> core/api/font_api
  -> screen_api 当前截图缓存
```

字库以 `shared_ptr` 不可变快照保存，识字线程取得快照后不持有字库表锁。设置或追加字库时先
构建新快照，再替换指定索引；追加同一索引的写操作串行执行，避免并发追加丢数据。

每个字形保存任意宽高的完整点阵，按行压缩成 `uint64_t`。加载时从完整点阵中选取纵向 11 位
低碰撞特征；识字区域也建立 `0` 到 `2047` 的坐标桶。特征只负责缩小候选范围，最终匹配始终
比较完整点阵中的缺失点和多余点。`fontFindStr` 使用完整识字结果，`fontFindStrFast` 只搜索
目标标签索引，两者共用同一套最终点阵比较。

## 脚本 UI 路径

```text
Lua HostApi
  -> system_c_api / core/api/ui_api
  -> AndroidBridge JNI（:engine）
  -> App 主进程的 Dialog Overlay Service、HUD Service、WebView Activity
  -> EngineHttpServer 的 ui.event
  -> core/api/ui_api 会话队列
  -> Lua m.ui.waitEvent / m.hud.waitEvent / m.web.waitEvent
```

`libengine.so` 负责会话 ID、配置、事件队列、等待和 C ABI；App 主进程只负责 Android
View、窗口和 WebView。三个界面实现分别是：

- `ScriptDialogOverlayService`：WindowManager 原生提示、确认、输入、选择和表单；框外触摸穿透。
- `ScriptHudService`：脚本 HUD 悬浮层、按钮、拖动和定时关闭。
- `ScriptWebActivity`：HTML、WebView 和 JavaScript bridge。

脚本 UI 事件不会在 Android UI 线程执行 Lua。事件先通过本机 HTTP `ui.event` 写入
native 队列，再由脚本线程读取。完整 API 见 [脚本文档（界面）](脚本文档.md) 与
[Android 脚本 UI（实现）](ANDROID_脚本_UI.md)。
