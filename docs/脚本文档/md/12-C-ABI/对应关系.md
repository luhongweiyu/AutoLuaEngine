## `EngineImGuiApi` 是什么

`EngineImGuiApi` 是 `libengine.so` 内的 ImGui 子函数表。顶层 `EngineApi` 从 ABI 18 开始在
末尾提供 `getImGuiApi`，直接导出 `engine_getImGuiApi()` 返回同一张只读函数表：

```c
const EngineApi* api = engine_getApi();
const EngineImGuiApi* imguiApi = api->getImGuiApi();
```

当前 `EngineImGuiApi::abiVersion` 为 `1`。表内包含框架、窗口、布局、控件、表格、图片、
样式、图形和事件等待函数。Lua 的函数对象不能进入 C ABI，因此 Lua 绑定保存回调，并通过
`engine_imguiWaitEvent()` 消费统一事件后执行。JS、Go 和插件使用相同事件队列，但分别按自己
的运行时模型分发回调。

正常 Lua 调用链如下：

```text
imgui.createButton(...)
  -> Lua ImGui 绑定
  -> engine_imguiCreateButton(...)
  -> core/api/imgui_api
```

完整声明见 `core/imgui_c_api.h`；逐方法映射见“ImGui 完整对应关系”，脚本参数和返回值见
“ImGui”文档分类。

## `EngineDeviceApi` 是什么

`EngineDeviceApi` 是 `libengine.so` 内的一张设备函数表，不是 Lua 命名空间，也不是另一套实现。它由 `engine_getDeviceApi()` 返回：

```c
const EngineDeviceApi* device = engine_getDeviceApi();
const char* brand = device->getBrand();
```

`device->getBrand()` 与直接调用 `engine_getBrand()` 使用同一份底层实现。Lua HostApi 当前使用第二种写法中的直接 C ABI，因此正常 Lua 脚本的实际调用链是：

```text
m.getBrand() -> Lua HostApi -> engine_getBrand() -> Android
```

函数表只给 JS、Go 和外部插件 SO 使用：它们可先取得 `EngineDeviceApi*`，再按 `device->函数名()` 调用整组设备能力，并通过 `abiVersion` 判断版本兼容性。下面表格的第三列仅展示这种插件调用写法，不代表多了一套设备命令。

`engine_deviceLastError()` 与 `device->lastError()` 也是同一份设备错误信息的两种入口：前者是直接 C ABI，后者是函数表成员。

除 `m.getRunEnvType()`、`m.getDisplayInfo()`、`m.getSensorsInfo()` 和应用列表三个 JSON 接口外，
Lua 名称、直接导出和函数表成员名称保持一一对应。Lua 层会把 JSON 转成 table，并按脚本
API 约定处理无返回值命令；C ABI 则保留 `1`/`0` 状态码或原始 JSON 文本。

`import` 是动态 Java 对象桥，不对应单个固定 C ABI。它的公开入口和 Lua userdata
元方法仍由 `libengine.so` 注册，Java 层只执行 JVM 类加载与反射。

`m.thread`、`beginThread` 和 `Thread.newThread` 是 Lua 运行时语义，也不对应通用
C ABI；它们直接由 `libengine.so/runtime/lua` 的调度器实现。
